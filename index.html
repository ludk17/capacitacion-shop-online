Anatomía de una Petición HTTP

El estándar de comunicación entre Clientes (Programas/Navegadores) y Servidores (APIs)

El Request (Petición)

1. Método (Verbo)

Indica la acción que queremos realizar en el servidor.

GET (Leer) POST (Crear) PUT (Actualizar) DELETE (Borrar)
2. Endpoint & Parámetros

La dirección exacta (URL) a la que llamamos. Puede incluir Query Params (filtros después del ?) o Path Params (variables en la ruta).

GET https://api.escuela.com/alumnos/15?estado=activo
// Path Param (ID: 15) | Query Param (Filtro: activo)
3. Headers (Cabeceras)

Metadatos ocultos que dan contexto sobre la petición (Seguridad, formato de datos, idioma).

Authorization: Bearer eyJhbGci...
Content-Type: application/json
4. Body (Cuerpo)

La carga útil (datos) que enviamos al servidor. Se usa principalmente en POST y PUT. Formato estándar: JSON.

{
  "nombre": "Ana Torres",
  "curso": "Automatización"
}

El Response (Respuesta)

1. Status Code

Un número de 3 dígitos que indica cómo resultó la petición. Es lo primero que validamos en automatización.

200 OK Éxito general
201 Created Recurso creado
400 Bad Req Error del cliente
401 Unauth Falta Token
404 Not Found No existe
500 Server Err Fallo la API
2. Headers (Servidor)

Información adicional que el servidor nos devuelve sobre la respuesta (tiempo, tamaño, tipo de servidor).

Date: Tue, 03 Mar 2026 12:00:00 GMT
Content-Type: application/json
X-Powered-By: Express
3. Body (Datos devueltos)

La información que solicitamos o la confirmación de la acción. Aquí es donde hacemos nuestras aserciones principales (Tests).

{
  "id": 101,
  "nombre": "Ana Torres",
  "createdAt": "2026-03-03T12:00:00Z"
}

Misión del Tester:

Controlar exactamente qué enviamos en el Request y automatizar la validación de cada pieza del Response.
